# アプリケーション設計図 (v2)

このドキュメントは、ユーザー認証を含む新しい要件に基づいたWebアプリケーションの構成、機能、データフローについて記述します。

## 1. 全体構成

基本的な構成はv1と同様、ReactフロントエンドとNode.js (Express) バックエンドです。v2では、これに**Google認証**が加わり、ユーザーごとのデータ管理が中心となります。

- **フロントエンド**: Material-UI, React Router, `@mui/x-charts`, `d3.js` を引き続き使用。Google認証用のライブラリ (`react-google-login` や `@react-oauth/google` など) を追加。
- **バックエンド**: Express, SQLiteを引き続き使用。Google認証のトークンを検証するためのライブラリ (`google-auth-library` など) と、セッション管理のためのライブラリ (`express-session` など) を追加。

## 2. 機能要件

### 2.1. 初期バージョン（コア機能）

まず、アプリケーションの基盤となる以下の機能を実装します。

- **ユーザー認証**:
  - Googleアカウントによる新規登録とログイン。
  - ログアウト機能。
  - 認証状態はサーバーサイドのセッションで管理。

- **基本的なポートフォリオ管理 (CRUD)**:
  - ログインユーザーは、自身のポートフォリオ資産を管理できる。
    - **追加 (Create)**: 資産名、数量、取得日、取得単価などを入力して新しい資産を追加。
    - **閲覧 (Read)**: 資産一覧、総資産額、含み損益、資産配分（グラフ）を表示。
    - **更新 (Update)**: 既存の資産情報を編集（リバランス対応）。
    - **削除 (Delete)**: 資産を削除。
  - ポートフォリオデータは、ログインしているユーザーに紐付けて管理する。

- **公開プロフィール設定**:
  - ユーザーは自身のプロフィールページを持ち、ポートフォリオを「公開」または「非公開」に設定できる。
  - プロフィールには、自己紹介、各種SNSへのリンク、暗号資産ウォレットアドレス（寄付受け取り用）を登録できる。

- **ランキングページ (シンプル版)**:
  - ポートフォリオを「公開」設定にしているユーザーを対象に、総資産額のランキングを表示する。

### 2.2. 将来的な拡張機能

コア機能の実装後、以下の機能を追加してアプリケーションを強化します。

- **高度なポートフォリオ機能**:
  - **ポートフォリオ診断**: 資産配分のリスクなどを分析・評価する機能。
  - **リファレンス機能**: 他のハイパフォーマンスな公開ユーザーのポートフォリオ構成を参考にできる機能。
  - **配当金トラッキング**: 受け取った配当金を記録・集計。
  - **ベンチマーク比較**: パフォーマンスを市場平均と比較。

- **通知機能**:
  - **価格アラート**: 銘柄が指定価格に達したら通知。
  - **定期サマリー**: 週次/月次でパフォーマンスをメール等で通知。

- **利便性の向上**:
  - **CSVインポート/エクスポート**: 資産データを一括登録・出力。
  - **多通貨対応**: JPY, USDなど複数の通貨での資産管理に対応。

- **UI/UXの強化**:
  - **カスタムCSS**: ユーザーが自分の公開ページのデザインを簡易的にカスタマイズできる機能。

## 3. データフロー（認証あり）

### 3.1. Googleログインフロー

1.  **ユーザー**: フロントエンドで「Googleでログイン」ボタンをクリック。
2.  **フロントエンド**: Googleの認証画面にリダイレクト。ユーザーが承認すると、Googleは認証トークン（IDトークン）をフロントエンドに返す。
3.  **フロントエンド**: 受け取ったIDトークンを、バックエンドの`/auth/google`エンドポイントに送信。
4.  **バックエンド**:
    - IDトークンを受け取り、Googleのライブラリを使ってトークンが正当か検証。
    - 検証に成功したら、トークン内のユーザー情報（Google ID, 名前, アバターURLなど）を取得。
    - データベースの`users`テーブルで、このGoogle IDを持つユーザーを検索。
      - **存在する場合**: ユーザー情報を取得し、セッションを作成してログイン完了。
      - **存在しない場合**: 新しいユーザーとして`users`テーブルに登録し、セッションを作成してログイン完了。
    - ログイン成功のレスポンスをフロントエンドに返す。
5.  **フロントエンド**: ログイン成功を受け、ユーザーをダッシュボード（ポートフォリオページなど）にリダイレクト。

### 3.2. 認証が必要なAPIリクエスト

- ログイン後、フロントエンドからバックエンドへのAPIリクエストには、セッション情報（Cookieなど）が自動的に添付される。
- バックエンドは、ポートフォリオ取得などのリクエストを受けるたびに、セッションを検証してユーザーを特定し、そのユーザーに紐づくデータのみを返す。

## 4. データベーススキーマ (v2)

v1から、ユーザー管理と機能拡張のためにスキーマを更新する。

- **`users` (新規)**: ユーザー情報を格納。
  - `id`: INTEGER, PRIMARY KEY
  - `google_id`: TEXT, UNIQUE (Googleから提供される一意のID)
  - `name`: TEXT (Googleから取得した表示名)
  - `avatar_url`: TEXT (Googleから取得したアバター画像のURL)
  - `created_at`: DATETIME

- **`profiles` (更新)**: ユーザーごとの公開プロフィール情報を格納。
  - `id`: INTEGER, PRIMARY KEY
  - `user_id`: INTEGER, UNIQUE (users.idへの外部キー)
  - `bio`: TEXT (自己紹介)
  - `is_public`: BOOLEAN (ポートフォリオを公開するか)
  - `sns_youtube`: TEXT
  - `sns_twitter`: TEXT
  - `wallet_eth`: TEXT (イーサリアムウォレットアドレス)
  - `wallet_bnb`: TEXT (BNBウォレットアドレス)
  - `custom_css`: TEXT

- **`portfolios` (更新)**: `user_id`でユーザーに紐づける。
  - `id`: INTEGER, PRIMARY KEY
  - `user_id`: INTEGER (users.idへの外部キー)
  - `asset_name`: TEXT
  - `asset_type`: TEXT (株式, 投資信託など)
  - `asset_sector`: TEXT (業種など)
  - `quantity`: REAL (数量)
  - `acquisition_date`: DATE (取得日)
  - `acquisition_price`: REAL (取得単価)
  - `location`: TEXT (保管場所: e.g., 'SBI証券')

- **`portfolio_history` (新規・将来用)**: パフォーマンス計算のために、資産のスナップショットを定期的に保存。
  - `id`: INTEGER, PRIMARY KEY
  - `user_id`: INTEGER
  - `snapshot_date`: DATE
  - `total_value`: REAL
